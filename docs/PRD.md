## 1. Product summary

**HyCoach** is a web app that acts as a personal Hyrox coach. Users enter their race date, fitness level, time commitment, and goals. The app generates a personalized, adaptive training plan that recalibrates weekly based on logged workouts. A "Coach" persona (user-selectable from three styles) delivers the plan with reasoning and weekly notes.

**Tagline:** *Your personal Hyrox coach.*

**Core promise:** Tells the user the truth about what it takes to hit their goal — and adapts the plan when life gets in the way.

**Primary user (v1):** Beginners who just signed up for their first Hyrox + recreational gym-goers/runners adding Hyrox. Advanced athletes can use it but aren't the design target.

**Division (v1):** Singles only. Doubles, Relay, Pro deferred.

**Long-term positioning (informational, not v1 scope):** HyCoach as a brand can extend to other hybrid endurance disciplines (DEKA, Spartan, hybrid CrossFit, marathon+strength training) without rebrand. v1 is Hyrox-specific in copy, messaging, and assessment — but the underlying engine (Reality Check + plan generation + rule-based adaptation + Coach personas) is sport-agnostic.

## 2. The three product moments

These define HyCoach. Every feature exists to support them.

### Moment A — Reality Check (the lead)

Reality Check pushes back on **commitment-to-goal math**, not predicted outcomes. The math is deterministic, computed in app code before any Claude call.

**Math (minutes-based — improved from sessions-based):**

- Required training minutes by goal tier (rough, calibrated to typical Hyrox prep):

  | Goal | Required minutes |
  |---|---|
  | Just finish (first-timer) | 1500-2000 |
  | Sub-1:30 (intermediate) | 3000-4000 |
  | Sub-1:15 (advanced) | 4500-6000 |
  | Sub-1:00 (elite) | 6500+ |

- Available minutes = `weeks_until_race × days_per_week × min_per_session`
- If `available_minutes < required_minutes_min × 0.85`, the goal is mathematically infeasible.

The Reality Check verdict is generated by Claude (Sonnet 4.5) and *includes the math* in coach voice with concrete options:

> *"You've got 10 weeks, 3 days a week, 60 minutes per session. That's 1,800 minutes. Sub-90 typically takes 3,000+ minutes of structured training to build the engine. Your math doesn't add up — but it's fixable two ways. Bump to 5 sessions/week (gets you to 3,000). Or aim for 'just finish' on this race and chase sub-90 next time. Which?"*
>
> [Increase to 5 days/week] [Adjust goal to 'just finish'] [Continue anyway, I accept the risk]

**State machine** (Reality Check ↔ Plan Generation):

```
[Onboarding inputs captured]
         ↓
[Compute commitment math (deterministic, app code)]
         ↓
[Call Reality Check (Claude Sonnet 4.5)]
         ↓
   ┌─────┴─────┐
feasible    not feasible
   ↓           ↓
   ↓     [Show pushback UI]
   ↓           ↓
   ↓     User picks adjustment
   ↓           ↓
   ↓     ┌─────┴─────┐
   ↓     adjust days/goal   "Continue anyway"
   ↓     ↓                  ↓
   ↓     [Loop back to math + Reality Check]
   ↓                        ↓
   └────────────────────────┘
         ↓
[Call Plan Generation (Claude Sonnet 4.5)]
         ↓
[Save plan, render Dashboard]
```

Each adjustment re-runs the math + Reality Check. Plan Gen is only called when feasibility is confirmed OR user explicitly accepts risk.

### Moment B — Plan reveal with reasoning

After Reality Check is resolved, Claude generates the plan in a single call. Every week includes a "Why this week" reasoning block written in coach voice. Reasoning is generated *alongside* the plan in the same call, not separately.

Example reasoning for week 3:
> *"You logged a 26-min 5K and you've never lifted seriously. This week we hold running volume flat and add the first real strength session — Romanian deadlifts and goblet squats. Building the posterior chain you'll need for sled work in week 7."*

### Moment C — Adaptation

The plan adapts in two ways:

1. **Weekly auto-recalibration** (rule-based, runs on **first user login of any new week**; idempotent per `(planId, weekNumber)` via `weeks.recalibrated` flag):
   - Adjusts upcoming week's prescribed values per rules in §8
   - Writes to `adjustments` table for audit trail
   - Triggers a Claude (Haiku) call for the weekly Coach note
   - User sees: "New week. Coach has notes." → expand → see what changed and why

   **No cron.** First-login is the only trigger. If a user doesn't open the app for two weeks, the recalibration runs when they return — for the most recent week only.

2. **User-triggered regeneration** (Claude Sonnet 4.5):
   - Eligibility (all must be true):
     - At least 7 days since last regeneration (cooldown), AND
     - One of: ≥30% workouts missed in last 2 weeks, OR consistently under-hit (≥3 logs in 2 weeks at <80% prescribed), OR injury flag added/changed since last plan, OR user explicitly chose "I want to revisit my goal" in settings
   - Otherwise: button grayed out with tooltip explaining why
   - On click: confirmation modal explains scope (preserves logs, regenerates upcoming weeks only, does not touch past)

## 3. The Hyrox knowledge primer

The Coach must speak like someone who actually understands Hyrox — not ChatGPT-with-a-coat. This primer is composed into every system prompt sent to Claude.

The primer lives in the repo as `prompts/hyrox-knowledge.md`, separate from code, easy to iterate. Full draft is in companion file `HyCoach-Prompts-v1.md`.

**Contents (high-level):**

- **What Hyrox is**: 8 × 1km runs alternating with 8 stations in fixed order: ski erg → sled push → sled pull → burpee broad jumps → rowing → farmers carry → sandbag lunges → wall balls. Total ~7.7-8km running plus stations.

- **Time tiers (Singles)** for context only:
  - World class: <55min M / <60min W
  - Elite age-grouper: 60-70min
  - Strong recreational: 70-85min
  - Recreational: 85-100min
  - First-timer: 100min+

- **The limiter**: "Compromised running" — runs feel hard because of accumulated station fatigue. Plans should include hybrid sessions (station + immediate run) from week 3+ onwards.

- **Common technique failure points**: wall balls (quad fatigue + ball-handling), sled push (lung-crushing, not technique-light), burpee broad jumps (form breaks under fatigue), sandbag lunges (posterior chain weakness), farmers carry (grip).

- **Equipment substitutions**:
  - No sled → heavy backward sled drags / weighted lunges + heavy push presses
  - No wall ball → squat-to-press with dumbbell + medicine ball thruster
  - No ski erg → rowing or assault bike at high cadence
  - No rower → assault bike or stair mill
  - All substitutions flag: "Substitute exercise. Find a Hyrox-equipped gym for at least 2 sessions on the actual machine before race day."

- **Phase structure**:
  - Base (first 30-40%): aerobic capacity, technique, GPP strength
  - Build (middle 40%): hybrid sessions intensify, station-specific work, threshold runs
  - Peak (next 15-20%): race-pace simulations, full or half Hyrox sims
  - Taper (final 7-10 days): volume drops 40-50%, intensity preserved short-and-sharp
  - Race week: light shake-out, mobility, mental prep

- **Coach voice principles** (apply across all personas):
  - Reference real Hyrox terminology
  - Reference user's actual logged data
  - Be specific to *this user's situation*; never generic
  - Never say "as an AI"
  - No emojis in coach output

## 4. User flows

### 4.1 First-time user flow

```
Landing page
  ↓ [Get started]
Google OAuth (Convex Auth)
  ↓
Onboarding sequence:
  1. Race date picker
     - Past date → blocked, error
     - <4 weeks away → warning panel; routes to detailed-only path
       (no skip-ahead allowed; Reality Check is stricter)
     - 4-52 weeks → standard path
     - >52 weeks → cap plan at 24 weeks; tell user we'll start the plan
       when they're 24 weeks out. Until then, they get a "preparation
       primer" placeholder (out of v1 scope; just a static page).
     - **Considering path (added v3.1, refined v3.1.1):** the date step
       presents a mutually-exclusive segmented choice: "I have a race
       date" or "I'm not registered yet". The picker shows a date input;
       the considering option shows four quick-pick chips (12 / 16 / 20
       / 24 weeks; date snapped to the next Saturday) with the chosen
       chip highlighted in green. Switching modes clears the other
       side's data — exactly one of `raceDate` (typed) or
       (`selectedWeeks` + computed `raceDate`) is captured at any time.
       Persists as `races.registrationStatus = "registered" |
       "considering"`. The plan engine still requires a date — this
       just makes the date easy to pick and tells Reality Check the
       date is flexible.
  2. Division: Singles only (Doubles disabled, "Coming in v2" tag)
  3. Goal: "Just finish" OR "Target finish time" (hh:mm:ss)
  4. Time commitment: days/week + minutes/session
  5. Fitness rating: Beginner / Intermediate / Advanced
  6. Adaptive depth question: "Want 2 minutes for a more accurate plan, or skip ahead?"
     - Skip → proceed with rating-only calibration
     - Detailed → 8-10 question assessment (next step)
  7. Detailed assessment (conditional):
     - 5K run pace (or "I don't know")
     - Push-up max
     - Estimated deadlift max (or "I don't lift")
     - Pull-up max
     - Past Hyrox attempts (yes/no, what time)
     - Current weekly running volume
     - Current weekly strength frequency
     - Years of consistent training
  8. Equipment access: sled, wall ball, ski erg, rower (yes/no per item)
  9. Coach persona selection: Honest / Encourager / Operator
     Each shown with one-line description and example phrase
  ↓
[Compute commitment math (deterministic, app code)]
  ↓
Reality Check (Convex action → Claude Sonnet 4.5)
  - Inputs: math result + all onboarding data
  - Output: { feasible: bool, verdict: string, suggestions: [...] }
  - State machine per §2 Moment A
  ↓
Plan generation (Convex action → Claude Sonnet 4.5)
  - Loading state with intentional copy ("Coach is reviewing your inputs...
    designing your weeks... finalizing your first session.")
  - Inputs: full profile + accepted goal + Hyrox knowledge primer + persona
  - Output: structured JSON plan via Anthropic tool-use (forced schema)
  - Server-side Zod validation
  - On parse failure: retry once with stricter prompt
  - On second failure: error UI ("Coach is having an off day. Try again
    in a moment."); log to Convex; do not persist invalid data
  ↓
Dashboard (first view)
  - Today's workout card (or "Start tomorrow" if today is rest)
  - Weeks until race counter
  - Welcome from Coach (persona-flavored, references actual plan)
```

### 4.2 Returning user flow

```
Sign in (auto-resume if session valid)
  ↓
Background checks (run as Convex queries on Dashboard mount):
  - Is user's current week's `recalibrated` flag false AND it's a new week?
    → call recalibration mutation:
       a) run rule-based adjustment engine
       b) write changes to `adjustments` table
       c) call Claude Haiku for weekly recap note
       d) set `weeks.recalibrated = true`
  - Has user gone 7+ days with no logs while workouts were prescribed?
    → show "I'm back" prompt before normal Dashboard
  ↓
Dashboard
  - Today's workout card (hero)
  - Daily/Weekly toggle (default daily)
  - This week's progress: streak, completion %
  - "Your plan" link → full plan view
  - "Coach has notes" banner if a new weekly recap was generated
```

### 4.3 Workout flow

Today's workout card on Dashboard shows:
- Workout type (Run / Strength / Hyrox / Hybrid / Rest)
- Title (e.g., "Threshold run" or "Lower body strength")
- Duration
- Prescribed exercises with sets/reps/weight/time
- "Why this workout" expandable section (short reasoning)

Two ways to log:

**(a) Quick "Mark done"** — single button click:
- Sets `workouts.status = "done"`, creates `workoutLog` with `quickDone: true, completedAt: now`
- Logs prescribed values as actuals (assumed to match)
- Animates a confirmation; updates streak + completion % live

**(b) "Log details"** — expandable form:
- Per-exercise: actual reps/sets/weight/time
- RPE slider (1-10)
- Notes textarea
- Submit → `workoutLog` with full data; `quickDone: false`

A third option: **"Skip"** button (less prominent, in expandable menu):
- Sets `workouts.status = "skipped"` — explicit user action
- Optional reason (free text)
- No `workoutLog` created

A workout is "missed" (computed at query time) if `status === "scheduled"` AND `scheduledDate < today` AND no `workoutLog` exists.

Coach micro-message after logging if RPE ≥ 9 or ≤ 3, or actuals deviate >20% from prescribed.

### 4.4 Missed week flow

Triggered when user signs in and: no logs in last 7+ days AND workouts were prescribed in that period.

"I'm back" modal (full screen on mobile, modal on desktop):
- "We haven't seen you in N days. What happened?"
- Two options:
  - **Forfeit & continue**: missed workouts marked `skipped`, plan continues from today, adaptation engine reduces volume on current week
  - **Pause & resume**: plan paused, user picks new "ready" date, all future workouts shift; if shift pushes past race date, prompt user to update race date

### 4.5 Race day & post-race flow

On race day, Dashboard shows special "Race day" card instead of regular workout card. Card content: motivational message in coach voice, race-day checklist (sleep, eat, mobility, mental cue).

After the race: "Log your race" form — finish time, splits (optional, per station), reflection textarea. On submit:
- Save `raceResult`
- Mark `races.status = "completed"`
- Trigger Claude Haiku for coach reflection (persona-flavored)
- Show post-race screen with reflection + two CTAs: "Set up next race" / "Take a break"

### 4.6 Race abandonment flow

Settings → "Manage race" → "Abandon this race":
- Confirmation modal: "This pauses your plan. You can set a new race anytime; your training history is preserved."
- Sets `races.status = "abandoned"`, `plans.status = "archived"`
- Dashboard shows "No active race" empty state with "Set up a new race" CTA
- All historical data (plans, logs, adjustments) preserved

### 4.7 Settings (lightweight in v1)

- Change coach persona (applies forward only — historical notes not rewritten)
- Edit profile (changes to fitness inputs trigger optional regeneration prompt, eligibility temporarily granted)
- Manage race (abandon, change date)
- Toggle dark mode (persisted to `profiles` so it follows the user across devices)
- Sign out
- Strava/Garmin connect (shown but disabled, "v2" tag)

## 6. Coach personas

Three personas, user-selectable at onboarding, switchable in settings. **Persona is a prompt parameter** — same logic, different voice. Persona changes apply forward only; historical notes preserved.

| Persona | Voice | Example: "You missed 3 sessions" |
|---|---|---|
| **The Honest Coach** *(default)* | Direct, warm, no sugarcoating. Treats user like an adult. | "You missed three sessions. Here's how we recover." |
| **The Encourager** | Supportive, growth-mindset, celebrates resets. | "Three off, three to come. You're still in this — let's reset." |
| **The Operator** | Intense, no-nonsense, race-day mindset. | "Three missed. Race won't wait. Lock in." |

Persona-specific tonal rules live in `prompts/personas/*.md` (see companion file).

## 8. Adaptation engine (rule-based)

Pure TypeScript logic running inside a Convex action. No LLM calls in this engine itself; the LLM is called separately for the recap note that explains the changes.

**When it runs:**
- On first user login of any new week (Dashboard mount checks `weeks.recalibrated`)
- Idempotent: checks `weeks.recalibrated` before running; sets `true` after
- No cron — first-login is the sole trigger

**Inputs:** Last week's prescribed workouts + `workoutLogs` for those workouts.

**Rules (v1, intentionally simple):**

1. **Missed workouts** (no log within 2 days of scheduled date AND not explicitly skipped):
   - 1 missed → no change
   - 2 missed → reduce next week's volume by 10%
   - 3+ missed → reduce next week's intensity by 15% AND volume by 10%; flag week as deload

2. **Skipped workouts** (user explicitly skipped):
   - Treated like missed for adaptation purposes, but logged separately

3. **RPE consistently low (avg < 5 across week's logs):**
   - Increase next week's intensity by 10% (within phase ceiling)

4. **RPE consistently high (avg > 8):**
   - Decrease next week's intensity by 10%; add deload notes for the week

5. **Under-hit prescribed numbers (actuals < 80% of prescribed across 3+ exercises in a week):**
   - Hold intensity flat; flag in coach note for honest acknowledgment

6. **Over-hit prescribed numbers (actuals > 110% in 3+ exercises):**
   - Increase next week's intensity by 5%

Every adjustment writes a row to `adjustments` for audit trail. After adjustments are computed, a single Claude Haiku call generates the weekly Coach note explaining what changed and why, in the user's selected persona voice.

## 9. Claude API integration

### 9.1 Architecture

**Hard rule: All Claude calls run inside Convex actions, server-side.** The frontend never touches the Anthropic API directly. The API key (`ANTHROPIC_API_KEY`) lives in Convex environment variables and is never exposed to the client.

Each call is wrapped in a Convex action that:
1. Reads inputs from Convex DB
2. Composes system prompt = `[Hyrox knowledge primer] + [Persona definition] + [Call-specific instructions]`
3. Calls Anthropic API via official SDK
4. Validates response (Zod schema for structured outputs)
5. Persists results to Convex DB
6. Returns minimum result needed by frontend

### 9.2 Calls and models

| Call | Model | When | Notes |
|---|---|---|---|
| Reality Check | claude-sonnet-4-6 | Onboarding, after math computed | Tone + math sensitive |
| Plan Generation | claude-sonnet-4-6 | After Reality Check resolved | Hardest reasoning; tool-use forces schema |
| Plan Regeneration | claude-sonnet-4-6 | User-triggered, eligibility-gated | Same call as Plan Generation |
| Weekly Recap | claude-haiku-4-5-20251001 | First login of new week | Short output; cost-sensitive at scale |
| Race Reflection | claude-haiku-4-5-20251001 | After race result logged | Short, voice-driven |

(Strings are aligned to code as of v3.1.2. Re-verify against Anthropic docs at each model upgrade; these change.)

### 9.3 Structured output handling

- All structured-output calls use **Anthropic tool-use** with `tool_choice: {type: "tool", name: "submit_X"}` and a JSON Schema for the input
- **Server-side Zod validation** of the tool-use input before persisting
- The call's prompt MUST instruct the model to return **tool-use only** — no prose, preamble, or commentary before, after, or alongside the tool call. Prose preamble is the dominant cause of empty/truncated tool inputs (the model exhausts `max_tokens` on text and then emits `{}` as tool input). See `prompts/plan-generation.md` `<output_protocol>` block for the canonical wording.
- On Zod validation failure: retry once with stricter prompt addition (e.g., "Your previous output failed validation: <error>. Output exactly the schema.")
- If the API response has `stop_reason: "max_tokens"` and Zod fails: log distinctly as an output-budget overflow ("Plan generation exceeded output budget before valid tool input"). The retry path runs as normal but operators should treat this as a prompt-engineering signal, not a token-cap signal — bumping `max_tokens` does not help if the model is writing prose.
- On second failure: log structural metadata (stop_reason, usage, content block types, input keys, shape of `weeks` — never values or user data), return user-facing error UX ("Coach is having an off day. Try again in a moment."), do not persist
- Streaming is **not** used in v1 — non-streaming with intentional loading copy is acceptable for 2–5 minute waits (Plan Generation is the slowest; Reality Check is ~5–15s)

### 9.4 Rate limiting

- Reality Check: server-side throttle, max 10 calls per user per onboarding session
- Plan Generation: gated by Reality Check completion + cooldown
- Plan Regeneration: cooldown enforced via `plans.lastRegenAt`
- Weekly Recap: idempotent via `weeks.recalibrated`; can't double-fire
- Race Reflection: only callable when `races.status` transitions to `completed`

### 9.5 Cost estimate (informational)

Rough per-user costs for v1.0 (back-of-envelope):
- Plan generation: ~$0.50-$2 per call
- Reality Check: ~$0.10-$0.30 per call
- Weekly Recap: ~$0.01-$0.05 per call (Haiku)
- Total per user across a 12-week plan: ~$1-$4

For 100 active users: ~$100-$400/month in API costs. Not blocking for v0.5/v1.0; revisit when scaling.

## 14. Critical edge cases (consolidated)

- Past race date → block with error
- Race <4 weeks away → warn + force detailed assessment + stricter Reality Check
- Race >52 weeks away → cap plan at 24 weeks; tell user we'll start the plan when they're 24 weeks out (placeholder page until then)
- User hasn't booked a race (`registrationStatus === "considering"`) → onboarding chips set a target date; Reality Check should prefer `extend_timeline` as one of its suggestions when the math is infeasible, since the date is flexible. (Added v3.1)
- User changes coach persona mid-plan → applies forward only
- User changes profile (e.g., adds injury) → prompt to regenerate; eligibility temporarily granted
- Claude returns invalid JSON → retry once → if still bad, error UX (no silent fallback)
- User logs workout for a workout already marked done → overwrite log, single source of truth
- User user signs in but onboarding incomplete → resume onboarding from last completed step (track `onboardingStep` in profiles)
- User abandons race → status = abandoned, all data preserved
- User races, then sets up next race → existing plan archived, new race + plan created
- Two browser tabs trying to recalibrate same week → idempotency on `weeks.recalibrated` prevents double-run
- Convex action timeout on slow Claude call → empirically Plan Generation requires up to ~3–4 minutes (Sonnet 4.6 emits ~83 tok/s; full plans use most of the 16k output budget). Set the Anthropic SDK `timeout` to **300s** and the SDK `maxRetries: 0` to prevent silent retry cascades that can exceed Convex's ~10-min platform cap. Application-level retry on Zod validation failure stays at exactly one attempt with a stricter user-message suffix.

## 16. Test cases (for v0.5 acceptance)

Manual test cases, run on staging before declaring v0.5 ready:

**Reality Check pushback (must trigger):**
- 4 weeks until race + 2 days/wk + 60min/session + sub-90 → infeasible, suggest 5 days OR finish goal
- 8 weeks until race + 3 days/wk + 45min/session + sub-75 → infeasible
- 6 weeks until race + 4 days/wk + 60min/session + sub-1:30 → infeasible (1440 min vs ~3000 needed)

**Reality Check pass (must NOT trigger):**
- 14 weeks until race + 4 days/wk + 60min/session + finish → feasible
- 20 weeks until race + 5 days/wk + 60min/session + sub-1:30 → feasible (6000 min, well above 3000-4000)
- 16 weeks until race + 4 days/wk + 75min/session + sub-1:30 → feasible (4800 min)

**Plan generation:**
- Beginner profile, no equipment except dumbbells → plan substitutes for sled, wall ball, ski erg, rower; flags clearly
- Advanced profile, sub-1:15 goal, 5 days/wk → plan includes hybrid sessions from week 3+

**Auth:**
- Google OAuth → signs in, profile created, lands on onboarding
- Refresh mid-onboarding → resumes from last step
- Sign out → clears session, returns to landing
