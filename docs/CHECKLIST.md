# HyCoach build checklist

This file is the **ordered, atomic source of truth** for build progression. The `/build-next` slash command reads this. Mark items `[x]` when done. Do not skip items.

When all items in a release are done, ship the release, then move to the next release section.

**Release strategy:** v0.5 ships to production first. v0.7, v1.0, and v1.1 are built on top of the deployed v0.5 — Convex hot-reloads schema changes, Vercel handles previews. Users on the live v0.5 will get newer features as each release is deployed. No re-platforming between releases.

**Reference:** Every item below is derived from `docs/PRD.md`. When in doubt, re-read the relevant PRD section. Numbers like "(PRD §4.1)" point you there.

## v1.0 — "Coach adapts with you"

**Promise:** Plan adapts when life happens.

**Reads from:** `docs/PRD.md` §2 Moment C, §4.2 (returning user flow), §4.4 (missed week), §8 (adaptation engine), §11 (v1.0 release notes).

### Phase K — Rule-based adaptation engine
- [ ] **K1.** Build `convex/lib/adaptationEngine.ts` per PRD §8: pure-function module that takes last week's prescribed + logs, returns array of `Adjustment` objects (workout-level changes with field/before/after).
- [ ] **K2.** Implement Rule 1 (missed workouts): 1 missed → no change, 2 missed → -10% volume, 3+ missed → -15% intensity AND -10% volume (deload flag)
- [ ] **K3.** Implement Rule 2 (skipped workouts): treated like missed for adaptation
- [ ] **K4.** Implement Rule 3 (avg RPE < 5): +10% intensity, within phase ceiling
- [ ] **K5.** Implement Rule 4 (avg RPE > 8): -10% intensity, add deload notes
- [ ] **K6.** Implement Rule 5 (under-hit prescribed in 3+ exercises): hold intensity flat, flag for coach note
- [ ] **K7.** Implement Rule 6 (over-hit prescribed in 3+ exercises): +5% intensity
- [ ] **K8.** Write unit tests for the adaptation engine in `convex/lib/__tests__/adaptationEngine.test.ts`. Cover all 6 rules + boundary cases. This is the only place v1 needs unit tests — pure logic with high stakes.

### Phase L — Weekly recalibration trigger
- [ ] **L1.** Add `weeks.recalibrated: boolean` and `weeks.recalibratedAt?: number` to Convex schema (already in PRD §7, just confirm it landed in B1)
- [ ] **L2.** Build Convex action `convex/actions/recalibrateWeek.ts`: checks `weeks.recalibrated` for current week; if false and it's a new week, runs `adaptationEngine`, writes adjustments to DB, calls weekly recap action, sets `recalibrated: true`. Idempotent.
- [ ] **L3.** Build Convex query helper `getCurrentWeek(planId)`: returns the week whose date range contains today
- [ ] **L4.** Trigger `recalibrateWeek` from Dashboard mount: useEffect that calls the action if conditions match. No cron — first-login is the sole trigger per PRD §8.
- [ ] **L5.** Apply adjustments to upcoming workouts in DB: update `exercises.prescribed` values per the adjustment; preserve audit trail in `adjustments` table

### Phase M — Weekly recap (Claude Haiku)
- [ ] **M1.** Write Convex action `convex/actions/weeklyRecap.ts`: composes prompt with `WEEKLY_RECAP` from `_prompts.ts` + persona; calls Anthropic Haiku 4.5; writes `weeks.coachNote` for the new current week. Plain text output, no tool-use.
- [ ] **M2.** Add "Coach has notes" banner to Dashboard when current week has a freshly-generated coach note (i.e., `recalibratedAt` is recent and user hasn't dismissed)
- [ ] **M3.** Build banner expand UI: shows the coach note + lists adjustments made (from `adjustments` table)

### Phase N — User-triggered regeneration
- [ ] **N1.** Build Convex query `canRegenerate(userId)` per PRD §2 Moment C: returns `{eligible: boolean, reason: string}`. Eligibility per PRD: 7-day cooldown AND (≥30% missed in 2 weeks OR consistent under-hit OR new injury OR explicit user intent).
- [ ] **N2.** Add "Regenerate plan" button to Dashboard / Settings; disabled state with tooltip explaining why per `canRegenerate` reason
- [ ] **N3.** Confirmation modal: explains scope (preserves logs, regenerates upcoming weeks only, does not touch past)
- [ ] **N4.** Build Convex action `convex/actions/regeneratePlan.ts`: archives current upcoming weeks, calls plan generation with full profile + log history context, persists new upcoming weeks, increments `plans.version`, sets `lastRegenAt`
- [ ] **N5.** Verify regeneration preserves past weeks (status `done`) and only replaces upcoming/current

### Phase O — Missed-week handling
- [ ] **O1.** Add detection logic in Dashboard mount: if user has 7+ days since last log AND workouts were prescribed in that period, show "I'm back" prompt before normal Dashboard
- [ ] **O2.** Build "I'm back" modal per PRD §4.4: "We haven't seen you in N days. What happened?" with two CTAs
- [ ] **O3.** Implement "Forfeit & continue": mutation marks missed workouts as `skipped`, runs adaptation engine immediately, plan resumes from today
- [ ] **O4.** Implement "Pause & resume": mutation pauses plan (new field `plans.pausedAt`), user picks "ready" date, all future workouts shift by the gap; if shift pushes past race date, prompt user to update race date
- [ ] **O5.** Add `plans.pausedAt?: number` field to schema if not already present from PRD §7

### Phase P — Progress page
- [ ] **P1.** Add Progress page route + navigation (header link from Dashboard)
- [ ] **P2.** Estimated finish-time chart (Recharts): plot estimated finish time over completed weeks. Pull "estimate" from a simple model — best-week pace projected to race conditions. Document the formula in `convex/lib/finishTimeEstimate.ts`.
- [ ] **P3.** Completion rate stat: % of prescribed workouts completed across the plan
- [ ] **P4.** Weekly volume chart: minutes trained per week (running + strength + hybrid stacked)
- [ ] **P5.** Streak history: current streak + longest streak this plan
- [ ] **P6.** Progress page responsive at 375px

### Phase Q — Adjustments visible in Plan View
- [ ] **Q1.** In Plan View, show adjustments inline per week: small badge "Adapted: -10% volume" with hover/click for details from `adjustments` table
- [ ] **Q2.** Highlight changed exercises in upcoming weeks (visual diff indicator)

## v2 (planning, not building)

After v1.1 ships and you have real users, plan v2 from the data you've collected. Candidate features (from PRD §10 OUT-of-v1 list):

- Doubles, Relay, Pro divisions
- Strava / Garmin auto-import
- Video station tutorials
- Community / social / leaderboards
- Payments / paid tier
- Email / push notifications
- Multiple concurrent plans
- Manual plan editing
- Nutrition guidance
- Native iOS / Android apps
